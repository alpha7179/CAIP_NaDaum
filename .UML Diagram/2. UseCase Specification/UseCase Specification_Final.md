# 유스케이스 명세서 최종 정리본

본 문서는 `UseCase Specification` 폴더에 정리된 최신 유스케이스 명세서를 최종 기준으로 삼아, 최종 보고서에 사용할 수 있도록 재구성한 문서입니다. UC-01, UC-02, UC-03의 명칭, Actor, include/extend 관계, 기본 흐름, 대안 흐름은 해당 폴더의 명세서를 우선 기준으로 정리하고, UML 다이어그램은 이 기준과 맞도록 표현을 조정하였습니다.

## 핵심 유스케이스 선정

| 구분 | 유스케이스 | 선정 이유 |
|---|---|---|
| UC-01 | 음성 대화를 통한 감정 기록 | 사용자의 음성 입력, STT 변환, 개인정보 마스킹, 멀티모달 감정 분석, AI 꼬리물기 질문, 정서 일기 및 상담용 문서 생성이 모두 포함되는 서비스의 중심 흐름입니다. |
| UC-02 | 종합 위험도 평가 및 안전망 연결 | 대화 중 또는 대화 종료 후 위험도를 평가하고, 저위험·중위험·고위험에 따라 안전 프로토콜을 수행하는 핵심 분기 흐름입니다. |
| UC-03 | 정서 일기 및 상담 자료 확인 | UC-01에서 생성된 정서 일기와 상담용 구조화 자료를 사용자가 회고·확인하고, 필요 시 상담사에게 공유·수정·삭제하는 흐름입니다. |

## UC-01. 음성 대화를 통한 감정 기록

### 기본 정보

| 항목 | 내용 |
|---|---|
| Use Case ID | UC-01 |
| Use Case 명 | 음성 대화를 통한 감정 기록 |
| 설명 | 사용자가 AI와 음성 대화를 통해 자신의 내면을 탐색하면, 시스템이 사전처리, 기록, 안전 프로토콜, 상호작용 흐름을 통해 멀티모달 감정 분석을 수행하고, 대화 종료 시 정서 일기와 상담용 구조화 문서를 자동 생성하여 DB에 저장합니다. |
| Primary Actor | 일반 사용자 |
| Secondary Actor | 외부 전문 상담 기관 |
| Precondition | 사용자가 로그인되어 있고, 음성 녹음 권한에 동의한 상태입니다. |
| Postcondition | 정서 일기와 상담용 구조화 문서가 DB에 저장되고, 사용자에게 하루 감정 요약이 제공됩니다. |

### 기본 성공 시나리오

1. (사용자 인터페이스 / User) 사용자는 홈 화면에서 "대화하기"를 선택합니다.
2. (사용자 인터페이스 / System) 시스템은 마이크를 활성화하고 새로운 대화 세션을 시작합니다.
3. (사용자 인터페이스 / User) 사용자는 오늘의 감정, 사건, 고민을 음성으로 말합니다.
4. (사전처리 / System) 시스템은 STT 기능을 통해 사용자의 음성을 실시간 텍스트로 변환합니다. 이 단계는 `<<include>>` STT 변환 흐름에 해당합니다.
5. (사전처리 / System) 시스템은 KoELECTRA NER 기반 개인정보 마스킹 모듈을 통해 이름, 장소, 날짜, 연락처 등 민감 정보를 비식별화합니다. 이 단계는 `<<include>>` 개인정보 마스킹 흐름에 해당합니다.
6. (정서 분석 / System) 시스템은 텍스트 문맥 감정 분석과 음성 프로소디 분석을 병렬로 수행합니다. 텍스트 분석은 7가지 감정과 강도를 산출하고, 음성 분석은 떨림, 피치, 에너지 등 비언어 지표를 산출합니다.
7. (정서 분석 / System) 시스템은 두 분석 결과를 캘리브레이션하여 종합 감정 지표를 생성합니다. 이 단계는 `<<include>>` 멀티모달 감정 분석 흐름에 해당합니다.
8. (안전 프로토콜 / System) 시스템은 감정 강도, 고위험 키워드, 대화 맥락을 기준으로 실시간 위험도를 판단합니다. 고위험으로 판단되면 UC-02 종합 위험도 평가 및 안전망 연결 흐름으로 확장됩니다.
9. (상호작용 / System) 시스템은 감정 지표와 대화 맥락을 기반으로 공감형 꼬리물기 질문을 생성하고, 필요 시 TTS 음성 응답으로 제공합니다.
10. (사용자 인터페이스 / User) 사용자는 AI 응답을 듣고 추가 답변을 이어갑니다.
11. (상호작용 / User/System) 사용자와 시스템은 상황 파악, 감정 탐색, 생각 탐색, 패턴 연결, 마무리의 5단계 구조에 따라 3~10단계를 반복합니다.
12. (사용자 인터페이스 / User) 사용자는 "대화 끝내기"를 선택합니다.
13. (기록 생성 / System) 시스템은 전체 대화 내용과 분석 결과를 바탕으로 사용자용 정서 일기와 상담용 구조화 문서를 병렬 생성합니다.
14. (기록 저장 / System) 시스템은 생성된 두 기록물을 DB에 저장합니다. 이 단계는 `<<include>>` 이중 기록물 자동 생성 흐름에 해당합니다.
15. (사용자 인터페이스 / User) 사용자는 하루 감정 요약, 정서 일기, 상담용 문서 생성 결과를 확인합니다.

### 대안 시나리오

#### 4a. 음성 인식 실패

1. (System) 음성이 정상적으로 인식되지 않으면 시스템은 재시도 안내를 표시합니다.
2. (System) 음성 인식이 반복해서 실패하면 시스템은 텍스트 입력 모드로 전환합니다.
3. (User/System) 사용자가 텍스트 입력을 완료하면 시스템은 기본 흐름의 개인정보 마스킹 단계로 복귀합니다.

#### 8a. 실시간 고위험 신호 감지

1. (System) 자살, 자해, 극단 선택 등 고위험 키워드 또는 임계치 초과 감정 지표가 감지되면 시스템은 대화 루프를 즉시 중단합니다.
2. (System) 시스템은 UC-02 종합 위험도 평가 및 안전망 연결 흐름을 실행합니다.
3. (System) UC-02 수행 이후에도 가능한 범위의 대화 내용은 정서 일기와 상담용 문서로 저장합니다.

### 관련 유스케이스

| 관계 | 대상 | 설명 |
|---|---|---|
| `<<include>>` | STT 변환 | 음성 입력을 분석 가능한 텍스트로 변환합니다. |
| `<<include>>` | 개인정보 마스킹 | 이름, 장소, 날짜, 연락처 등 민감 정보를 비식별화합니다. |
| `<<include>>` | 멀티모달 감정 분석 | 텍스트 감정과 음성 프로소디를 결합하여 종합 감정 지표를 생성합니다. |
| `<<include>>` | 이중 기록물 자동 생성 | 일기 생성과 상담용 문서 생성을 포함하여 대화 결과 기록물을 함께 생성합니다. |
| `<<extend>>` | UC-02 종합 위험도 평가 및 안전망 연결 | 대화 중 고위험 신호가 감지되거나 대화 종료 후 위험도 평가가 필요한 경우 확장됩니다. |

## UC-02. 종합 위험도 평가 및 안전망 연결

### 기본 정보

| 항목 | 내용 |
|---|---|
| Use Case ID | UC-02 |
| Use Case 명 | 종합 위험도 평가 및 안전망 연결 |
| 설명 | 대화 중 실시간 고위험 신호가 감지되거나 대화 종료 시점에, 시스템이 텍스트 감정 수치, 음성 프로소디, 고위험 키워드, 자살위험 탐지 결과를 통합하여 위험도를 판단하고 단계별 안전 프로토콜을 수행합니다. |
| Primary Actor | 일반 사용자 |
| Secondary Actor | 외부 전문 상담 기관, 보호자 |
| Precondition | UC-01 대화 세션이 종료되었거나, 대화 중 실시간 고위험 신호가 감지된 상태입니다. |
| Postcondition | 위험도 레벨에 따른 개입이 완료되고, 위기 이벤트 로그가 DB에 저장됩니다. 중·고위험의 경우 안부 확인 스케줄이 등록될 수 있으며, 고위험의 경우 보호자 알림 및 전문기관 연결이 시도됩니다. |

### 기본 성공 시나리오

1. (System) 시스템은 UC-01 종료 또는 대화 중 고위험 신호 감지를 트리거로 위험도 평가를 시작합니다.
2. (System) 시스템은 텍스트 감정 수치, 음성 프로소디 지표, 고위험 키워드 탐지 결과를 통합합니다.
3. (System) 시스템은 KoELECTRA 기반 자살위험 탐지 모델을 활용하여 종합 위험도를 평가합니다.
4. (System) 시스템은 위험도 판단 결과를 저위험, 중위험, 고위험 중 하나로 분기합니다.
5. (System) 저위험이면 시스템은 일상 감정 관리 팁을 제공하고 지속 기록을 권장합니다.
6. (System) 중위험이면 시스템은 상담 서비스 이용을 추천하고 24시간 후 안부 확인 메시지 발송을 예약합니다.
7. (System) 고위험이면 시스템은 긴급 대응 모드를 활성화하고 전문 심리 상담 기관 연결 UI를 전면 표시합니다.
8. (System) 고위험 상황에서 시스템은 자살예방상담 1393, 정신건강위기상담 1577-0199 등 긴급 연락처를 제공합니다.
9. (System) 보호자 연락처가 등록되어 있으면 시스템은 보호자 알림을 발송하고, 등록되어 있지 않으면 보호자 등록 안내를 표시합니다.
10. (System) 시스템은 중·고위험에 대해 지속 모니터링 필요 여부를 판단하고 후속 안부 확인 스케줄을 등록합니다.
11. (System) 시스템은 위험도 판단 결과, 사용자 반응, 안내 내용, 알림 발송 여부를 이벤트 로그 DB에 저장합니다.
12. (User/System) 사용자는 안내를 확인하고 홈 화면 또는 일기 화면으로 복귀합니다.

### 대안 시나리오

#### 1a. 대화 중 실시간 고위험 신호 감지

1. (System) UC-01 대화 루프 중 고위험 키워드 또는 감정 임계치 초과가 감지되면 시스템은 대화를 즉시 중단합니다.
2. (System) 시스템은 고위험 분기 단계부터 즉시 실행합니다.

#### 5H-a. 사용자가 전문기관 연결을 거부

1. (User) 사용자가 전문기관 연결을 거부합니다.
2. (System) 시스템은 강제 연결은 수행하지 않지만 긴급 연락처와 즉시 도움 요청 안내를 화면에 유지합니다.
3. (System) 시스템은 거부 사실과 안내 제공 여부를 이벤트 로그에 저장합니다.

#### 5H-b. 보호자 연락처 미등록

1. (System) 보호자 연락처가 없으면 시스템은 알림 발송을 보류합니다.
2. (System) 시스템은 보호자 등록을 요청하는 안내를 표시하고, 긴급 상담 연락처를 우선 제공합니다.

### 관련 유스케이스

| 관계 | 대상 | 설명 |
|---|---|---|
| `<<extend>>` | UC-01 음성 대화를 통한 감정 기록 | UC-01 대화 중 또는 종료 후 위험도 판단이 필요한 경우 실행됩니다. |
| `<<extend>>` | 일상 팁 제공 | 저위험 분기에서 실행됩니다. |
| `<<extend>>` | 상담 서비스 추천 | 중위험 분기에서 실행됩니다. |
| `<<extend>>` | 상담 연결 및 보호자 연결 | 고위험 분기에서 실행됩니다. |
| `<<extend>>` | 안부 알림 및 재대화 유도 | 중·고위험 분기에서 후속 모니터링을 위해 실행됩니다. |

## UC-03. 정서 일기 및 상담 자료 확인

### 기본 정보

| 항목 | 내용 |
|---|---|
| Use Case ID | UC-03 |
| Use Case 명 | 정서 일기 회고 및 상담 자료 확인·공유 |
| 설명 | 사용자가 일기 화면에 접속하면 시스템이 정서 일기 목록을 제공하고, 사용자는 일기 상세를 열람한 뒤 상담용 구조화 문서를 확인·공유하거나 일기를 수정·삭제할 수 있습니다. |
| Primary Actor | 일반 사용자 |
| Secondary Actor | 심리 상담사, 외부 전문 상담 기관 |
| Precondition | UC-01이 완료되어 정서 일기와 상담용 구조화 문서가 1건 이상 저장된 상태입니다. |
| Postcondition | 사용자가 자신의 감정 변화를 회고하고, 필요 시 상담사에게 구조화된 사전 자료를 전달합니다. 수정·삭제 결과와 관련 로그는 DB에 반영됩니다. |

### 기본 성공 시나리오

1. (사용자 인터페이스 / User) 사용자는 홈 화면 또는 하단 내비게이션에서 "일기"에 접속합니다.
2. (기록 아카이브 / System) 시스템은 사용자의 이전 기록 존재 여부를 확인합니다. 기록이 없으면 저장된 기록 없음 대안 흐름으로 이동합니다.
3. (기록 아카이브 / System) 시스템은 DB에서 정서 일기 목록을 날짜순으로 조회합니다. 이 단계는 `<<include>>` 일기 목록 조회 흐름에 해당합니다.
4. (사용자 인터페이스 / System) 시스템은 조회된 일기 목록을 카드 형태로 표시합니다.
5. (사용자 인터페이스 / User) 사용자는 월별, 감정별, 키워드별 필터 조건을 선택할 수 있습니다. 이 단계는 `<<extend>>` 일기 목록 필터링 흐름에 해당합니다.
6. (기록 아카이브 / System) 필터 조건이 있으면 시스템은 조건에 맞는 일기 목록을 재조회하여 표시합니다.
7. (사용자 인터페이스 / User) 사용자는 열람할 일기를 선택합니다.
8. (사용자 인터페이스 / System) 시스템은 선택한 일기의 이야기체 본문, 감정 태그, 감정 수치 시각화, 생성 날짜를 표시합니다.
9. (사용자 인터페이스 / User) 사용자는 상담 문서 확인, 상담 자료 공유, 수정, 삭제, 목록 복귀 중 필요한 행동을 선택합니다.
10. (개인정보 보호 / System) 사용자가 공유하기를 선택하면 시스템은 공유 전 개인정보 마스킹 상태와 사용자 동의를 다시 확인합니다.
11. (기록 아카이브 / System) 시스템은 상담 기관 공유용 포맷으로 구조화 문서를 내보냅니다. 이 단계는 `<<extend>>` 상담 자료 공유 흐름에 해당합니다.
12. (사용자 인터페이스 / User) 사용자는 열람 또는 공유를 마치고 홈 화면 또는 일기 목록으로 복귀합니다.

### 대안 시나리오

#### 2a. 저장된 기록 없음

1. (System) 아카이브가 비어 있으면 시스템은 "아직 대화 기록이 없어요. 대화를 시작해보세요." 안내를 표시합니다.
2. (System) 시스템은 사용자를 UC-01 음성 대화를 통한 감정 기록 흐름으로 유도합니다.

#### 5a. 필터링 조건 미선택

1. (User/System) 사용자가 필터를 선택하지 않으면 시스템은 전체 일기 목록 상태를 유지합니다.
2. (User) 사용자는 전체 목록에서 열람할 일기를 선택합니다.

#### 9a. 일기 수정

1. (User) 사용자는 일기 상세 화면에서 "수정"을 선택합니다.
2. (User) 사용자는 본문을 수정한 뒤 "저장"을 선택합니다.
3. (System) 시스템은 수정 내용을 DB에 반영하고 수정 로그를 저장합니다.

#### 9b. 일기 삭제

1. (User) 사용자는 일기 상세 화면에서 "삭제"를 선택합니다.
2. (System) 시스템은 삭제하면 복구할 수 없다는 경고를 표시합니다.
3. (User) 사용자는 일기만 삭제할지, 연결된 상담용 문서까지 함께 삭제할지 선택합니다.
4. (System) 시스템은 선택한 범위에 따라 삭제를 수행하고 삭제 로그를 저장합니다.

### 관련 유스케이스

| 관계 | 대상 | 설명 |
|---|---|---|
| `<<include>>` | 일기 목록 조회 | 사용자의 저장 기록을 날짜순으로 조회합니다. |
| `<<extend>>` | 일기 목록 필터링 | 월별, 감정별, 키워드별 조건으로 기록을 필터링합니다. |
| `<<extend>>` | 상담 문서 확인 | 선택한 일기에 연결된 상담용 구조화 자료를 표시합니다. |
| `<<extend>>` | 상담 자료 공유 | 상담사 또는 외부 전문 상담 기관에 전달 가능한 포맷으로 내보냅니다. |
| `<<extend>>` | 일기 수정 및 삭제 | 사용자가 기록을 직접 관리하고 변경 이력을 DB에 반영합니다. |
| 선행 유스케이스 | UC-01 음성 대화를 통한 감정 기록 | UC-03은 UC-01에서 생성된 기록물이 있어야 수행됩니다. |
